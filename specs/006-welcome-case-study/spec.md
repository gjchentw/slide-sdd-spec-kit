# 功能規格：新增 Welcome 專案 SDD 實戰案例投影片

**功能分支**: `006-welcome-case-study`
**建立日期**: 2026-02-18
**狀態**: 草案
**輸入**: 使用者描述: "請探索 @Welcome/ 目錄，這是一個運用 spec-kit 完成的專案，請產生一組投影片，以 Welcome 專案作為 spec-kit 的實際運用案例，說明符合 SDD 開發流程的地方"

## 使用者場景與測試 *(強制項目)*

### 使用者故事 1 - 概覽 Welcome 專案的 SDD 結構 (優先級: P1)

作為學員，我希望看到 Welcome 專案如何組織其規格與實作，以便理解 SDD 在真實專案中的目錄結構。

**為何以此優先級**: 這是展示「規格即代碼」最直觀的方式，讓使用者看到實踐結果。

**獨立測試**: 確認投影片包含 Welcome 專案的 `specs/` 目錄截圖或結構說明，並對應到 SDD 的各個階段。

**驗收場景**:

1. **假設** 投影片已載入, **當** 使用者翻到 Welcome 案例章節, **則** 應顯示 Welcome 專案的 `specs/` 資料夾結構及其內容（如 `001-spigot-init`, `012-i18n-support`）。

---

### 使用者故事 2 - 深入分析單一功能的開發歷程 (優先級: P1)

作為開發者，我希望以 Welcome 的 `012-i18n-support` 為例，看到從 `spec.md` 到 `plan.md` 再到代碼的轉化過程。

**為何以此優先級**: 具體的案例（國際化支援）比抽象的概念更能解釋 SDD 的運作細節。

**獨立測試**: 確認投影片展示了 `012-i18n-support` 的規格要求（多語系支援）與最終實作結果（LanguageManager）的關聯。

**驗收場景**:

1. **假設** 正在介紹開發歷程, **當** 點擊案例分析, **則** 應展示該功能的規格截圖與對應的代碼/文件產出。

---

### 使用者故事 3 - 驗證 Welcome 專案的憲章執行 (優先級: P2)

作為架構師，我希望看到 Welcome 專案如何透過 `GEMINI.md` 維護其技術憲章，並在各個功能中貫徹。

**為何以此優先級**: 展示 SDD 如何確保大型專案在長期演進中保持架構的一致性。

**獨立測試**: 確認投影片列出 Welcome 的核心憲章原則（如外部 Markdown、正體中文優先）並說明如何自動化檢查。

---

## 需求 *(強制項目)*

### 功能需求

- **FR-001**: 必須新增一個獨立的 Markdown 檔案 `slides/05-welcome-case-study.md`。
- **FR-002**: 內容必須包含 Welcome 專案的背景介紹（Spigot 插件開發）。
- **FR-003**: 內容必須包含「實戰案例 1：初始化架構 (001-spigot-init)」，展示基礎建置的 SDD 流程。
- **FR-004**: 內容必須包含「實戰案例 2：國際化支援 (012-i18n-support)」，展示複雜功能的規格拆解。
- **FR-005**: 必須使用 Mermaid 流程圖展示 Welcome 專案的 CI/CD 流程與 SDD 的結合。
- **FR-006**: 必須在 `index.html` 中載入此新章節，置於「規格迭代與分支策略」之後。

### 憲章合規性 (Constitution Compliance)

- **CC-001**: 投影片內容「必須」儲存在外部 `.md` 檔案中。
- **CC-002**: 技術圖表「必須」使用 Mermaid.js 語法。
- **CC-003**: 佈局「必須」針對 21:9 (1920x1080) 進行優化。
- **CC-004**: 部署「必須」透過 Actions 自動化至 GitHub Pages。
- **CC-005**: 內容與文件「必須」使用正體中文。

## 成功準則 *(強制項目)*

### 可衡量結果

- **SC-001**: 投影片完整呈現 Welcome 專案至少兩個功能的完整 SDD 循環（Spec -> Plan -> Tasks -> Implement）。
- **SC-002**: 展示 Welcome 專案的 `GEMINI.md` 與 `specs/` 的真實文件結構。
- **SC-003**: Mermaid 圖表能正確表達 SDD 如何與 Gradle/Spigot 的開發流程整合。
