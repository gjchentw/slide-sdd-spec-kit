# 功能規格：新增 Spec Kit 規格迭代與分支策略投影片

**功能分支**: `005-add-sdd-iteration-intro`
**建立日期**: 2026-02-18
**狀態**: 草案
**輸入**: 使用者描述：新增介紹 spec-kit 的規格迭代與分支策略，涵蓋自動建立分支、規格與代碼共存、版本控制一致性，以及如何處理規格不確定性（拒絕猜測、探索性生成、檢查清單、持續演進）。

## 使用者場景與測試 *(強制項目)*

### 使用者故事 1 - 了解分支策略 (優先級: P1)

作為開發者，我希望了解 Spec Kit 如何利用 Git 分支管理規格與代碼，以便理解「膠囊化」的開發概念。

**為何以此優先級**: 這是 SDD 版本控制的核心機制，確保規格即代碼。

**獨立測試**: 導航至分支策略章節，確認包含「自動綁定」、「規格代碼共存」、「版本一致性」的說明。

**驗收場景**:

1. **假設** 投影片已載入，**當** 使用者翻頁至分支策略，**則** 應顯示 Git 分支如何封裝規格資產與生成代碼的說明。

### 使用者故事 2 - 掌握不確定性處理 (優先級: P1)

作為 PM 或架構師，我希望了解 SDD 如何處理模糊不清的需求，以便在規格不完整時仍能推進開發。

**為何以此優先級**: 解答常見疑慮，展示 SDD 的靈活性。

**獨立測試**: 確認投影片包含處理不確定性的四種機制：拒絕猜測、探索性生成、品質攔截、持續演進。

---

## 需求 *(強制項目)*

### 功能需求

- **FR-001**: 必須新增一個獨立的 Markdown 檔案（如 `slides/04-sdd-iteration.md`）。
- **FR-002**: 內容必須包含「分支策略」：
  - 自動建立與綁定分支
  - 規格與代碼共存 (膠囊化概念)
  - 版本控制的一致性 (真理來源與實作結果對齊)
- **FR-003**: 內容必須包含「不確定性處理」：
  - AI 拒絕猜測 (標記 [NEEDS CLARIFICATION])
  - 探索性生成 (Stable enough vs Complete)
  - 品質攔截 (/speckit.checklist)
  - 持續演進 (Living Documentation)
- **FR-004**: 必須包含 Mermaid 流程圖，視覺化展示 Git 分支如何支援規格迭代。
- **FR-005**: 投影片佈局必須符合 21:9 寬螢幕設計規範。
- **FR-006**: 必須在 `index.html` 中載入此新章節，置於 SDD 指令流程之後。

### 憲章合規性 (Constitution Compliance)

- **CC-001**: 投影片內容「必須」儲存在外部 `.md` 檔案中。
- **CC-002**: 技術圖表「必須」使用 Mermaid.js 語法。
- **CC-003**: 佈局「必須」針對 21:9 (1920x1080) 進行優化。
- **CC-004**: 部署「必須」透過 Actions 自動化至 GitHub Pages。
- **CC-005**: 內容與文件「必須」使用正體中文。

## 成功準則 *(強制項目)*

### 可衡量結果

- **SC-001**: 投影片完整涵蓋分支策略的 3 個重點與不確定性處理的 4 個機制。
- **SC-002**: Mermaid 流程圖正確渲染，清晰展示規格迭代循環。
- **SC-003**: 內容文字清晰易讀，無排版溢出。

### 邊界案例

- **文字量過大**: 若「不確定性處理」的四點說明文字過多，應拆分為獨立頁面或使用重點列表。
