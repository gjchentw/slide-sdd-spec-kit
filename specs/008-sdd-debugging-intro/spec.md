## 使用者場景與測試 *(強制項目)*

### 使用者故事 1 - 建立正確的 SDD 除錯心態 (優先級: P1)

作為開發者，我希望了解為何不能直接修復代碼，而是要回頭修改規格。

**為何以此優先級**: 這是改變開發習慣最困難但也最重要的一步，確立「規格即代碼」的真理來源。

**獨立測試**: 確認投影片包含「修復配方，而非修復蛋糕」的核心原則，以及 Bug 即規格缺失的概念。

**驗收場景**:

1. **假設** 投影片載入, **當** 翻至「規格除錯」章節, **則** 應首先看到強調「不要當水管工，要當建築師」的比喻。

---

### 使用者故事 2 - 掌握 SDD 修復循環的操作步驟 (優先級: P1)

作為 RD/QA，我希望知道在已上線環境下（如 005 已上線，修復 003）的具體操作流程。

**為何以此優先級**: 提供可執行的 SOP，確保團隊在修復 Bug 時不會破壞 SDD 的一致性。

**獨立測試**: 確認投影片包含四個標準步驟：建立修復環境 -> 更新規格 -> 執行 SDD 循環 -> 驗證合併。

**驗收場景**:

1. **假設** 正在閱讀投影片, **當** 查看流程圖, **則** 應清楚看到從 checkout 舊功能分支到 PR 合併回主幹的完整路徑。
2. **假設** 查看詳細步驟, **則** 應包含「新增失敗測試 (Red Phase)」的關鍵環節。

## 需求 *(強制項目)*

### 功能需求

- **FR-001**: 必須新增一個獨立的 Markdown 檔案 `slides/07-sdd-debugging.md`。
- **FR-002**: 內容必須包含 SDD 除錯的核心哲學：「Bug 的存在即代表規格或計畫的不完整」。
- **FR-003**: 內容必須包含四個修復步驟：
  1. **建立修復環境**: 從 main 切出修復分支 (e.g., `003-fix`)。
  2. **挑戰規格**: 在 `spec.md` 中新增邊界條件與失敗測試案例。
  3. **執行修復循環**: Update Plan -> Fail Test -> Fix Code。
  4. **驗證與合併**: 確保回歸測試通過並合併回 main。
- **FR-004**: 必須包含總結頁面，強調這樣做能留下「歷史教訓」在規格中。
